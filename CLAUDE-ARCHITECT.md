## System Architecture

### State Ownership

All authoritative game state lives in these modules. Never duplicate state across modules — read from the owner.

| Module | Owns |
|---|---|
| `ServerState` | Country owners, coins, manpower, occupied territory chains |
| `TileManager` | NPC tile occupancy (which NPC is on which tile) |
| `TileOwnershipManager` | Diplomatic tile occupation (which country controls which foreign tile) |
| `DiplomacyManager` | Wars, alliances, NAP timers |
| `UnitManager` | Per-player NPC list (for stacking/overlap detection) |
| `CountryNPCRegistry` | Per-country NPC index (for unowned countries still holding NPCs) |

### Data Flow

```
ServerState (SSOT for country/resource data)
    ↑ written by CountryManager (income ticks, player joins/leaves)
    ↑ written by TileOwnershipManager (conquest events)

TileManager (NPC occupancy)
    ↑ written by NPCSpawner (spawn), NPCMovementSystem (movement), NPCCombatSystem (death)

DiplomacyManager (war/alliance/NAP)
    ↑ written by DiplomacyServer (RemoteEvent handlers)
    → read by NPCCombatSystem (can this NPC attack?)
    → read by NPCMovementSystem (can this NPC path here?)
```

## Tile Identity: Two Systems

**Never mix these up — they serve different purposes.**

### TileId (authoritative identity)
- Attribute: `TileId` on each BasePart (set to `GetFullName()` at startup if not Studio-authored)
- Used for NPC occupancy: `TileManager.GlobalOccupiedTilesById[tileId]`
- Stable across gameplay; used for all game logic

### TileKey (positional, legacy)
- Format: `"x,z"` where x and z are `math.round(position)` — 1-stud grid precision
- Generated by `TileKeyUtils` — use this module everywhere, never inline the formula
- Still present for backward compatibility; new code should prefer TileId

### Lookup in TileAdjacencyManager
- `TileIdToPart[tileId]` — primary identity lookup
- `PositionToTiles["x,z"] = {BasePart, ...}` — spatial bucket (many-to-many, not unique)
- `getTileFromKey(k)` returns `nil` if >1 candidate in positional bucket (safe-fail, not wrong tile)

## Map Structure

```
Workspace
  └─ EuropeMap
       └─ [Country Model]  (e.g., "France", "Germany")
            └─ [Tile BaseParts]
```

Capital tiles are identified by name/attribute, not position. Country models are the unit of ownership. `TileManager` scans this structure once at startup; never scan it at runtime.

## NPC Ownership Model

Each NPC has an `NPCOwner` ObjectValue (child of the NPC model):
- **Player-owned:** `NPCOwner.Value = Player` — follows the player; cleaned up on `PlayerRemoving`
- **Country-owned:** `NPCOwner.Value = nil` — persists when no player owns the country; has `AutoRetaliate = true` attribute

When a player leaves, `CountryManager` transfers their NPCs to country-owned state. When a new player claims that country, they inherit all country-owned NPCs.

NPC templates cloned from: `ReplicatedStorage.AttackerNPC`, `ReplicatedStorage.DefenderNPC`.

CollectionService tag for all live NPCs: `"RegisteredNPC"`.

## Conquest Chain Pattern

When a country's capital is captured, ownership becomes `"Conquered_X"` (where X is the conquering country's name). These chains can nest: `"Conquered_Conquered_France"`.

Always resolve chains before any war/diplomacy check:
```lua
local trueOwner = ServerState.resolveTarget(countryName)
local trueOwnerOfB = ServerState.resolveTarget(countryB)
```

`DiplomacyServer` applies this to every incoming event. Skipping resolution = wrong war checks.

## RemoteEvents Reference

All remotes live in `ReplicatedStorage`. The server creates them if missing; clients wait for them.

| Remote | Direction | Purpose |
|---|---|---|
| `GetAvailableCountries` | Client→Server (Function) | Returns country list, `false` if already owned, `nil` if not ready |
| `SelectCountryEvent` | Client→Server | Player claims a country |
| `CountryClaimedEvent` | Server→All | Announces new country owner |
| `InitializationComplete` | Server→All/Player | Server init done; unblocks client retry loops |
| `NPCMoveToTileEvent` | Client→Server | Move player's NPC to clicked tile |
| `BuyAttackerEvent` | Client→Server | Purchase attacker NPC |
| `BuyDefenderEvent` | Client→Server | Purchase defender NPC |
| `DeclareWarEvent` | Client→Server | Declare war on a country |
| `SendRequestEvent` | Client→Server | Send alliance/NAP request |
| `BreakAllianceEvent` | Client→Server | Break an alliance |
| `DiplomacyRequest` | Server→Player | Incoming alliance/NAP offer |
| `DiplomacyResponse` | Client→Server | Accept/reject diplomacy offer |
| `DiplomacyReminderEvent` | Server→Player | Popup: war started, NAP expired, insufficient manpower |
| `GlobalNotificationEvent` | Server→All/Player | Toast notification |

## Game Lifecycle

### Server Startup (sequential)
1. `ServerState` initializes (synchronous, instant)
2. `TileManager` scans `EuropeMap`, builds tile lists and stats (yields ~several seconds)
3. `CountryManager` calculates manpower per country (weighted: 50% population, 35% area, 15% tile count → 5k–90k range), spawns starter NPC at each capital, starts coin (+50 every 10s) and manpower (+100 every 3s) income loops
4. `InitializationComplete` fires to all players

### Player Join
1. `Leaderstats` creates `Coins`, `Manpower`, `Country` values
2. `CountrySelectionScreen` polls `GetAvailableCountries` (0.1s retry, signal-driven wake on `InitializationComplete`)
3. Player claims country → `CountryManager.handleCountrySelection()` transfers NPCs, inherits coins/manpower/wars

### Player Leave
- Coins and manpower saved to `ServerState` (persists for next player)
- NPCs converted to country-owned (`NPCOwner.Value = nil`)
- Wars and occupied territories **are not cleared**
- Country marked "Unowned"; next player to claim inherits everything

## Diagnostics & Debugging

`src/server/Diagnostics.server.luau` — runtime debug tooling. Check here first when investigating adjacency or occupancy issues.

`src/server/Auto-Fix.server.luau` — startup self-healing for known data issues.

Long-running systems expose a `DEBUG_MODE` flag or runtime-toggleable attribute for logging. Toggle these before adding new print statements.
